
単純ではなかったこと

- MIを使うと，unparseが必要になる上，ターミナル上での表示とずれる
- MIを使わないと，ページャーや[y/n]など，余計なユーザ入力が必要となる
  - gdb -q の使用，yの明示的な入力
- Ctrl-Cの入力が必要な場面があるが，TSの sendText では送れない（別ボタンで実装）
- アドレスでブレークポイントする問題の場合，アドレスが異なってしまう
  - これは ASLRやPIEをオフにしても解決できない
- 補完やヒストリ機能はコードセル上では実行できない．(頑張って実装することは可能だが大変)
  - 独自のコマンド候補一覧を表示する機能は実装済み

# 確認プロンプトを出さない
set confirm off
# 出力をページ分割しない
set pagination off
# debuginfoの自動ダウンロードを無効化
set debuginfod enabled off
# 画面高さ制限を無効化
set height 0
# 行の折り返しを無効化
set width 0

set debuginfod enabled off
とかは環境によって，出たり出なかったりする．上記の ~/.gdbinit を要求せざるを得ない．

=========
GDBノートブック
- 仮説立案→予測→実験→検証，という体系的なデバッグプロセスの記述・記録支援
- 失敗の記録は消去せず，なぜその仮説が棄却されたのかを記述させる
- simulated interactive debug 機能もあると良い
- Jupyter が科学計算で成功した理由は、 実験 結果 解釈 可視化
- バグパターンの蓄積
- LLM にデバッグ手順をレビューしてもらえる．flailing検出など．
- データセット
  - bugbench
